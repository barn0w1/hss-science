## 2. How browser flows work (SSR lifecycle recap)

This section reinforces and extends the baseline `research.md` with implementation-specific detail.

### 2.1 The fundamental contract

1. **Browser** hits Kratos init endpoint (`/self-service/<flow>/browser`) directly. Kratos creates a flow, sets the anti-CSRF cookie, and `303` redirects to the configured UI URL (e.g. `https://accounts.hss-science.org/login?flow=<uuid>`).
2. **SSR loader** receives the request with `?flow=<uuid>` and forwards the browser's `Cookie` header to Kratos to fetch the flow object via `GET /self-service/<flow>/flows?id=<uuid>`.
3. **SSR route** renders the HTML form dynamically from `flow.ui.nodes`. The form `action` attribute is set to exactly `flow.ui.action` (which points back to Kratos). The browser submits natively.
4. **Kratos** processes the form submission and either:
   - Returns `303` to the same UI route with the same/new `?flow=<uuid>` (validation error) â€” loader re-fetches and re-renders with error messages.
   - Returns `303` to the success/return URL and sets the session cookie.

**No React Router `action` function is involved in the credential submission path.** Form submission goes directly to Kratos.

### 2.2 Cookie forwarding is mandatory

The anti-CSRF cookie set by Kratos in step 1 must be echoed back by the browser on every subsequent request to Kratos. When the SSR loader calls `getLoginFlow(...)`, the flow ID alone is insufficient â€” Kratos validates the CSRF cookie too.

**SSR pattern:**

```ts
const cookie = request.headers.get("cookie") ?? undefined;
const flow = await kratosClient.getLoginFlow({ id: flowId, cookie });
```

The SDK maps the `cookie` parameter to the `Cookie` HTTP request header.

### 2.3 When no `?flow` param is present

The loader should redirect the browser to the Kratos init endpoint, which will create the flow, set cookies, and redirect back.

```ts
if (!flowId) {
  return redirect(`${KRATOS_BROWSER_URL}/self-service/login/browser`);
}
```

The browser flow init URL is on the **browser-facing Kratos URL**, not the internal one, because the browser must follow that redirect and receive the cookie from Kratos.

## 3. The `@ory/kratos-client-fetch` SDK

This is a key difference from the sample app, which uses `@ory/client` (axios-based). The fetch-based SDK has different error and redirect behavior.

### 3.1 SDK initialization

```ts
import { FrontendApi, Configuration } from "@ory/kratos-client-fetch";

// Internal URL â€” for server-to-server calls in loaders
const kratosInternal = new FrontendApi(
  new Configuration({
    basePath: process.env.KRATOS_PUBLIC_URL, // e.g. http://kratos-public.identity.svc.cluster.local
  })
);
```

The `KRATOS_BROWSER_URL` (the external URL) is used only for constructing redirect targets that the **browser** will navigate to.

### 3.2 Error behavior â€” `ResponseError`

The SDK throws `ResponseError` for **any non-2xx response** (including `303`, `4xx`, `5xx`). This is unlike axios, which only throws on network errors by default.

```ts
import { ResponseError } from "@ory/kratos-client-fetch";

try {
  const flow = await kratosInternal.getLoginFlow({ id: flowId, cookie });
} catch (err) {
  if (err instanceof ResponseError) {
    const status = err.response.status;
    const body = await err.response.json().catch(() => null);
    // handle by status code
  }
}
```

**Important:** `err.response` is a fetch `Response` object. The body stream can only be read once â€” use `.clone()` or read it once and store.

### 3.3 Redirect behavior

By default, the SDK uses `fetch` with `redirect: "follow"`. A Kratos `303` on flow init or completion will be auto-followed and you may receive an unexpected `200` from the redirect target. This is **acceptable for loader-side flow fetches** (which never trigger Kratos session-setting redirects), but requires care if you ever call update/submit endpoints server-side.

For any case where you need to capture a `303` and its `Set-Cookie`, use `initOverrides` to disable auto-follow:

```ts
const raw = await kratosInternal.someFlowRaw(params, {
  redirect: "manual",
});
```

With `redirect: "manual"`, a `303` surfaces as a `ResponseError` with `err.response.status === 303`, and you can read `err.response.headers.get("location")` and `err.response.headers.get("set-cookie")`.

### 3.4 Raw methods for header access

Each SDK method has a companion `*Raw` version that returns an `ApiResponse` object with `.raw` (the native `Response`) and `.value()` (the parsed typed body):

```ts
const apiResponse = await kratosInternal.getLoginFlowRaw({ id: flowId, cookie });
const setCookie = apiResponse.raw.headers.get("set-cookie");
const flow = await apiResponse.value(); // typed LoginFlow
```

Use raw methods when you need to forward `Set-Cookie` from the Kratos response to the browser's response headers.

### 3.5 Session check â€” `toSession`

The equivalent of `/sessions/whoami` is:

```ts
const session = await kratosInternal.toSession({ cookie });
```

This is used in the settings route guard (must be authenticated) and for the index route (redirect authenticated users away from login/registration).

## 4. `UiNode` type system in detail

The entire rendering model is driven by the `flow.ui` object. Understanding the node schema is critical.

### 4.1 `UiContainer` structure

```ts
interface UiContainer {
  action: string;       // POST target URL â€” always flow.ui.action
  method: string;       // "POST"
  messages?: UiText[];  // flow-level messages (e.g. "Invalid credentials")
  nodes: UiNode[];      // the ordered list of nodes to render
}
```

The `<form>` element is:
```html
<form action="{flow.ui.action}" method="{flow.ui.method}">
```

### 4.2 `UiNode` structure

```ts
interface UiNode {
  type: UiNodeTypeEnum;          // "input" | "text" | "img" | "a" | "script" | "div"
  group: UiNodeGroupEnum;        // which auth method this node belongs to
  attributes: UiNodeAttributes;  // discriminated union by node_type
  messages: UiText[];            // field-level validation messages
  meta: UiNodeMeta;              // { label?: UiText }
}
```

### 4.3 Node groups

Groups determine which authentication method a node belongs to:

| Group | Purpose |
|---|---|
| `default` | Always present: `csrf_token` hidden input, identifier fields shared across methods |
| `password` | Password credential fields |
| `oidc` | Social/OAuth2 provider buttons (one per provider) |
| `profile` | Identity trait fields (name, etc.) â€” driven by identity schema |
| `code` | Magic link / code-based auth fields |
| `totp` | TOTP (Google Authenticator) code input, QR code setup |
| `webauthn` | WebAuthn/hardware key fields and script |
| `passkey` | Passkey (FIDO2) fields and script |
| `lookup_secret` | Backup recovery codes |
| `identifier_first` | Identifier-first login (email input before method selection) |
| `captcha` | CAPTCHA challenge |
| `saml` | SAML SSO buttons |
| `link` | Recovery link method (legacy) |

**Key rule:** Nodes from the `default` group (especially `csrf_token`) must always be included in every form submission. In practice, render all `default` group nodes as hidden inputs in every form variant.

### 4.4 Node type discriminated union

`UiNodeAttributes` is a discriminated union on `node_type`:

```ts
type UiNodeAttributes =
  | ({ node_type: "input" } & UiNodeInputAttributes)
  | ({ node_type: "text" }  & UiNodeTextAttributes)
  | ({ node_type: "img" }   & UiNodeImageAttributes)
  | ({ node_type: "a" }     & UiNodeAnchorAttributes)
  | ({ node_type: "script" } & UiNodeScriptAttributes)
  | ({ node_type: "div" }   & UiNodeDivisionAttributes);
```

Discriminate using `node.attributes.node_type` or `node.type` (they are mirrors of each other).

### 4.5 Input node (`node_type: "input"`)

```ts
interface UiNodeInputAttributes {
  name: string;           // form field name â€” use as-is in <input name="...">
  type: UiNodeInputAttributesTypeEnum; // "text"|"password"|"email"|"hidden"|"submit"|"button"|"checkbox"|...
  value?: any;            // current value / preset value for hidden/submit inputs
  required?: boolean;
  disabled: boolean;
  label?: UiText;         // label text (preferred over node.meta.label for inputs)
  autocomplete?: string;
  maxlength?: number;
  pattern?: string;
  onclick?: string;       // deprecated; use onclickTrigger
  onclickTrigger?: string; // WebAuthn trigger name (e.g. "oryPasskeyLogin")
  onload?: string;        // deprecated; use onloadTrigger
  onloadTrigger?: string; // WebAuthn trigger name
  node_type: "input";
}
```

**Rendering rule:** For `type: "hidden"`, render as a hidden `<input>`. For `type: "submit"` or `type: "button"`, render as a button. For `type: "checkbox"`, render a checkbox. All other types render as standard `<input type="...">`.

**Value handling:** Always set the `value` attribute when `attributes.value` is present (this is especially critical for hidden fields like `csrf_token`, `method`, and OIDC provider identifiers).

### 4.6 Text node (`node_type: "text"`)

```ts
interface UiNodeTextAttributes {
  id: string;
  text: UiText;   // { id, text, type, context }
  node_type: "text";
}
```

Text nodes render displayable content, not form inputs. Used for:
- TOTP secret display (copy/paste friendly)
- Lookup/backup recovery code lists (download affordance helpful)
- Informational messages embedded in the flow

### 4.7 Image node (`node_type: "img"`)

```ts
interface UiNodeImageAttributes {
  id: string;
  src: string;   // usually a data URI (e.g. "data:image/png;base64,...")
  width: number;
  height: number;
  node_type: "img";
}
```

Used for TOTP QR code display during setup. Render as `<img src={attrs.src} width={attrs.width} height={attrs.height} alt={node.meta.label?.text} />`.

### 4.8 Anchor node (`node_type: "a"`)

```ts
interface UiNodeAnchorAttributes {
  id: string;
  href: string;     // destination URL
  title: UiText;    // label text
  node_type: "a";
}
```

Render as a link button. Used by Kratos for certain action links.

### 4.9 Script node (`node_type: "script"`)

```ts
interface UiNodeScriptAttributes {
  id: string;
  src: string;          // script URL (Kratos-served WebAuthn JS)
  type: string;         // "text/javascript"
  async: boolean;
  integrity: string;    // SRI hash â€” MUST be set exactly
  crossorigin: string;
  referrerpolicy: string;
  nonce: string;        // for CSP
  node_type: "script";
}
```

**Critical:** Script nodes must be rendered **outside the form**, typically at the bottom of the page. All attributes (especially `integrity`, `crossorigin`, `referrerpolicy`) must be set exactly as provided. Omitting `integrity` breaks SRI validation and may cause browser errors.

Script nodes appear when WebAuthn or Passkey methods are enabled. The script loaded from `src` registers the `window.__oryWebAuthnRegistration`, `window.__oryWebAuthnLogin`, `window.__oryPasskeyLogin`, etc. functions.

### 4.10 `onclickTrigger` and `onloadTrigger` (new in v26+)

These replace the deprecated `onclick`/`onload` string attributes. Instead of `eval()`-ing raw JS, the new triggers are string keys that map to well-known functions on the Ory WebAuthn script:

| Trigger key | Action |
|---|---|
| `oryWebAuthnRegistration` | Register a WebAuthn hardware key |
| `oryWebAuthnLogin` | Authenticate with WebAuthn |
| `oryPasskeyLogin` | Passkey login |
| `oryPasskeyLoginAutocompleteInit` | Initialize passkey autocomplete |
| `oryPasskeyRegistration` | Register a passkey |
| `oryPasskeySettingsRegistration` | Register passkey in settings |

**Rendering pattern (new trigger API):**

```tsx
// When rendering a submit button with onclickTrigger:
const handleClick = () => {
  if (attrs.onclickTrigger) {
    // Call the function provided by the Ory WebAuthn script
    const fn = (window as any).__ory?.[attrs.onclickTrigger];
    if (typeof fn === "function") fn();
  }
};
```

The `onloadTrigger` is invoked after the script loads:
```tsx
// After mounting, call onload triggers
useEffect(() => {
  nodes.forEach(node => {
    if (node.attributes.node_type === "input" && node.attributes.onloadTrigger) {
      const fn = (window as any).__ory?.[node.attributes.onloadTrigger];
      if (typeof fn === "function") fn();
    }
  });
}, [nodes]);
```

### 4.11 `UiText` (messages)

```ts
interface UiText {
  id: number;      // stable numeric ID â€” use for i18n key lookup
  text: string;    // English fallback text
  type: "info" | "error" | "success";
  context?: object; // template variables for custom formatting
}
```

Messages appear at two levels:
1. `flow.ui.messages[]` â€” flow-level (e.g. "Invalid credentials")
2. `node.messages[]` â€” field-level (e.g. "This value is required")

Both must be rendered. Style by `type`: errors in red, info in neutral, success in green.

## 5. React Router v7 SSR patterns

### 5.1 Loader pattern for Kratos flows

**Canonical loader for a flow route:**

```ts
import type { Route } from "./+types/login";
import { redirect } from "react-router";
import { FrontendApi, Configuration, ResponseError } from "@ory/kratos-client-fetch";

const kratosInternal = new FrontendApi(
  new Configuration({ basePath: process.env.KRATOS_PUBLIC_URL })
);

export async function loader({ request }: Route.LoaderArgs) {
  const url = new URL(request.url);
  const flowId = url.searchParams.get("flow");
  const cookie = request.headers.get("cookie") ?? undefined;

  // No flow ID â€” redirect browser to Kratos init endpoint
  if (!flowId) {
    return redirect(`${process.env.KRATOS_BROWSER_URL}/self-service/login/browser`);
  }

  try {
    const flow = await kratosInternal.getLoginFlow({ id: flowId, cookie });
    return { flow };
  } catch (err) {
    if (err instanceof ResponseError) {
      if (err.response.status === 410 || err.response.status === 404) {
        // Expired or not found â€” restart flow
        return redirect(`${process.env.KRATOS_BROWSER_URL}/self-service/login/browser`);
      }
      if (err.response.status === 403) {
        // Possibly requires re-auth (AAL step-up)
        const body = await err.response.json().catch(() => ({})) as any;
        if (body?.redirect_browser_to) {
          return redirect(body.redirect_browser_to);
        }
        return redirect(`${process.env.KRATOS_BROWSER_URL}/self-service/login/browser`);
      }
      if (err.response.status === 422) {
        // browser_location_change_required
        const body = await err.response.json().catch(() => ({})) as any;
        if (body?.redirect_browser_to) {
          return redirect(body.redirect_browser_to);
        }
      }
    }
    throw err; // Let ErrorBoundary handle unexpected errors
  }
}
```

### 5.2 Typed loader args

React Router v7 provides auto-generated types via `react-router typegen`. Always import from the generated `+types` module:

```ts
import type { Route } from "./+types/login";

export async function loader({ request }: Route.LoaderArgs) { ... }
export default function LoginPage({ loaderData }: Route.ComponentProps) { ... }
```

The `loaderData` type is automatically inferred from the loader return type.

### 5.3 Component rendering pattern

```tsx
export default function LoginPage({ loaderData }: Route.ComponentProps) {
  const { flow } = loaderData;
  return (
    <form action={flow.ui.action} method={flow.ui.method}>
      {/* Render nodes dynamically */}
      {flow.ui.nodes.map(node => (
        <FlowNode key={getNodeId(node)} node={node} />
      ))}
    </form>
  );
}
```

**No React Router `<Form>` component for the Kratos form.** Use a native `<form>` element so the browser submits directly to Kratos without React Router interception. If using React Router `<Form>` for other purposes on the same page, make sure it points to a different action (not Kratos).

### 5.4 Forwarding Set-Cookie from Kratos

When Kratos returns `Set-Cookie` headers during a flow fetch (e.g. when creating a verification flow server-side), forward them to the browser:

```ts
import { data } from "react-router";

const raw = await kratosInternal.getLoginFlowRaw({ id: flowId, cookie });
const flow = await raw.value();
const setCookie = raw.raw.headers.get("set-cookie");

const headers = new Headers();
if (setCookie) headers.append("Set-Cookie", setCookie);

return data({ flow }, { headers });
```

**Note:** For simple `getLoginFlow` / `getRegistrationFlow` calls, Kratos typically does not return new `Set-Cookie` on flow fetch (cookies were set at init time). Forwarding is most relevant when the SSR creates a new flow server-side (e.g., the login route creates a verification flow and redirects).

### 5.5 Error boundary

Export `ErrorBoundary` from route modules:

```tsx
import { isRouteErrorResponse, useRouteError } from "react-router";

export function ErrorBoundary() {
  const error = useRouteError();
  if (isRouteErrorResponse(error)) {
    return <ErrorCard status={error.status} message={error.data} />;
  }
  return <ErrorCard status={500} message="An unexpected error occurred." />;
}
```

### 5.6 Route guards (session check)

For protected routes (settings), verify the session in the loader:

```ts
try {
  await kratosInternal.toSession({ cookie });
} catch (err) {
  if (err instanceof ResponseError && err.response.status === 401) {
    return redirect(`${process.env.KRATOS_BROWSER_URL}/self-service/login/browser`);
  }
  if (err instanceof ResponseError && err.response.status === 403) {
    // AAL2 required
    return redirect(`${process.env.KRATOS_BROWSER_URL}/self-service/login/browser?aal=aal2`);
  }
  throw err;
}
```

For routes that should be inaccessible when authenticated (login, registration, recovery), check `toSession` and redirect to home/settings if a session exists.

## 6. Per-flow implementation details

### 6.1 Login (`/login`)

**Loader responsibilities:**
1. Read `?flow`, `?aal`, `?refresh`, `?return_to` from URL.
2. If no `?flow`: redirect to `/self-service/login/browser` with `aal`, `refresh`, `return_to` forwarded as query params.
3. Fetch `getLoginFlow({ id: flowId, cookie })`.
4. Check `flow.ui.messages` for message ID `4000010` (requires email verification before login) â†’ create a browser verification flow server-side and redirect to `/verification?flow=<id>&message=...`.
5. Return flow data.

**Special queries at init:**
- `?refresh=true` â€” force re-auth (updates `authenticated_at`)
- `?aal=aal2` â€” require second factor
- `?return_to=<url>` â€” post-success redirect

**Component responsibilities:**
- Render `default` group nodes (always, including `csrf_token` hidden).
- Render method groups separately:
  - `password` group: email/identifier + password + submit
  - `oidc` group: one button per provider (each is a submit with `name="provider"` and `value="<provider-id>"`)
  - `code` group (if code method enabled): identifier field + code input (two-step UX)
  - `passkey`/`webauthn` group: trigger buttons + script loading
- Show "Forgot password" link â†’ `recovery` init URL.
- Show "Sign up" link â†’ `registration` init URL.
- For AAL2 / refresh flows: show "Logout" link (fetch logout URL from `createBrowserLogoutFlow`).

**Node groups to render per submit path:**
- When submitting password: include `default` + `password` groups.
- When submitting OIDC: include `default` + `oidc` (specifically the clicked provider's submit node).
- Method dispatch correctness: Kratos uses the submitted `method` hidden field value to route. Only one method-specific `method=password` or `method=oidc` node should be submitted per form submission.

**Flow states:** `choose_method` | `sent_email` (code method) | `passed_challenge`

---

### 6.2 Registration (`/registration`)

**Loader responsibilities:**
1. Read `?flow`, `?return_to`, `?after_verification_return_to`, `?login_challenge`, `?organization`, `?identity_schema` from URL.
2. If no `?flow`: redirect to init endpoint forwarding all relevant query params.
3. Fetch `getRegistrationFlow({ id: flowId, cookie })`.
4. Return flow data.

**Component responsibilities:**
- Render `default` group (csrf_token always).
- `profile` group: dynamic identity trait fields â€” name, custom traits from schema. Never hardcode field names.
- `password` group: password input + submit.
- `oidc` group: provider buttons.
- `code` group: email field + submit (if code registration enabled).
- Show "Sign in" link â†’ `login` init URL.

**Post-registration branch:**
- If Kratos is configured with `session` after-hook: user lands on return URL with session cookie â†’ treat as logged in.
- Without session hook: Kratos redirects to login page â†’ user must log in.
- If email verification is required: Kratos redirects to `/verification`.

**Flow states:** `choose_method` | `sent_email` | `passed_challenge`

---

### 6.3 Settings (`/settings`)

**Loader responsibilities:**
1. Check session via `toSession({ cookie })` â€” redirect to login if no session.
2. Read `?flow`, `?return_to`.
3. If no `?flow`: redirect to `/self-service/settings/browser`.
4. Fetch `getSettingsFlow({ id: flowId, cookie })`.
5. Check `flow.state`: `show_form` = render; `success` = show success message.
6. Return flow data + session.

**Component responsibilities:**
Settings has multiple method sections. Each should be visually distinct but all submit to `flow.ui.action`:

- **Profile section** (`profile` group): trait fields + save button.
- **Password section** (`password` group): new password input + save button.
- **OIDC section** (`oidc` group): link/unlink provider buttons (OIDC submit nodes with different `value` fields for link/unlink actions).
- **TOTP section** (`totp` group): QR code (`img` node) + secret (`text` node) + confirmation code input + setup/unlink submit.
- **WebAuthn section** (`webauthn` group): register/remove security key flows.
- **Passkey section** (`passkey` group): similar to WebAuthn.
- **Lookup secrets section** (`lookup_secret` group): display backup codes (`text` node with list) + reveal/generate button.

**Privileged session handling:**
- Some operations (password change, OIDC link/unlink, MFA changes) require `authenticated_at` to be within `privileged_session_max_age`.
- Kratos returns `403` on the settings flow fetch with a `NeedsPrivilegedSessionError` body containing `redirect_browser_to`.
- Detect in loader and redirect to `redirect_browser_to` (which is a login flow with `refresh=true`).

**Settings flow state:**
- `state: "show_form"` â€” normal form display.
- `state: "success"` â€” a section was saved. Re-render the form showing a success indicator. Kratos does NOT redirect away on settings success for browser flows.

---

### 6.4 Recovery (`/recovery`)

**Loader responsibilities:**
1. Check for active session â†’ redirect to settings if already logged in (use `requireNoAuth` equivalent).
2. Read `?flow`, `?return_to`.
3. If no `?flow`: redirect to `/self-service/recovery/browser`.
4. Fetch `getRecoveryFlow({ id: flowId, cookie })`.
5. Return flow data.

**Component responsibilities (code method â€” two-step):**

**Step 1 â€” `choose_method` state:**
- Render `default` group (csrf_token).
- Render `code` or `link` group: email/address input + submit.
- Show "Back to sign in" link.

**Step 2 â€” `sent_email` state:**
- Render `default` group.
- Render `code` group: code input + submit.
- Render resend button (appears as a `code` group submit node with different value).
- Show informational message: "Check your email" (neutral, non-enumerating).

**Post-recovery:**
- On success, Kratos issues a privileged session and redirects to the settings flow (for password reset).
- `return_to` carries through automatically from recovery to settings.

**Non-enumeration principle:** The UI must NOT reveal whether an email address has an account. Messages must be worded neutrally (e.g. "If an account exists, a recovery code has been sent.").

---

### 6.5 Verification (`/verification`)

**Loader responsibilities:**
1. Read `?flow`, `?return_to`, `?message` from URL.
2. If no `?flow`: redirect to `/self-service/verification/browser`.
3. Fetch `getVerificationFlow({ id: flowId, cookie })`.
4. If `?message` query param is present (custom messages injected by login route when email verification is required): parse JSON and inject into `flow.ui.messages`.
5. Return flow data.

**Component responsibilities (code method â€” two-step):**

**Step 1 â€” `choose_method` state:**
- Render `default` group.
- Render `code` group: email/address input + submit.
- Show "Sign up" link.

**Step 2 â€” `sent_email` state:**
- Render code input + submit.
- Resend button.
- Neutral messaging.

**Step 3 â€” `passed_challenge` state:**
- Verification complete. Render success screen.
- Show "Continue" / navigate to app.

### 6.6 Logout (`/logout`)

Logout in a browser flow is a **two-step navigation**, not a form submission. There is no "logout flow" object with `ui.nodes`.

**Loader:**
```ts
export async function loader({ request }: Route.LoaderArgs) {
  const cookie = request.headers.get("cookie") ?? undefined;
  try {
    const logoutFlow = await kratosInternal.createBrowserLogoutFlow({ cookie });
    // logoutFlow = { logout_url: string, logout_token: string }
    // Redirect the browser to the logout_url (which is a Kratos-hosted GET endpoint)
    return redirect(logoutFlow.logout_url);
  } catch (err) {
    if (err instanceof ResponseError && err.response.status === 401) {
      // No session â€” already logged out
      return redirect("/login");
    }
    throw err;
  }
}
```

The `logout_url` already includes the session token. The browser navigates to it, Kratos terminates the session and redirects to the configured post-logout URL.

**Alternative pattern** (show confirmation first):
1. Loader fetches `createBrowserLogoutFlow` â†’ returns `{ logout_url, logout_token }` to component.
2. Component shows a "Do you want to sign out?" prompt.
3. User clicks "Yes" â†’ navigate browser to `logout_url`.

### 6.7 Error (`/error`)

**Loader:**
```ts
export async function loader({ request }: Route.LoaderArgs) {
  const url = new URL(request.url);
  const errorId = url.searchParams.get("id");
  // Also check for OAuth2-style error params
  const oauthError = url.searchParams.get("error");

  if (oauthError) {
    return {
      error: {
        id: oauthError,
        code: 599,
        message: url.searchParams.get("error_description") ?? oauthError,
        hint: url.searchParams.get("error_hint") ?? undefined,
      }
    };
  }

  if (!errorId) {
    return { error: { id: "unknown", message: "An unknown error occurred." } };
  }

  try {
    const flowError = await kratosInternal.getFlowError({ id: errorId });
    // flowError = { id, error: { code, message, reason, ... } }
    return { flowError };
  } catch (err) {
    return { error: { id: "fetch_failed", message: "Could not load error details." } };
  }
}
```

**Component:**
- Render `flowError.error.message` (and optionally `reason`, `debug` fields if available).
- Show a "Back" link using `document.referrer` or a default return URL.
- Never show raw stack traces in production.

## 7. Critical error handling matrix

| HTTP Status | Context | Loader action |
|---|---|---|
| `401` | `toSession`, `getLoginFlow` | Redirect to `/self-service/login/browser` |
| `403` | Settings flow fetch | Check `redirect_browser_to` (privileged session) â†’ redirect to login with `refresh=true` |
| `403` | AAL2 required | Redirect to `redirect_browser_to` (login with `aal=aal2`) |
| `404` | Any flow fetch | Flow not found â†’ restart (redirect to init) |
| `410` | Any flow fetch | Flow expired â†’ restart (redirect to init) |
| `422` | Any SDK call | `browser_location_change_required` â†’ redirect to `redirect_browser_to` |
| `303` (manual) | SDK call with `redirect: "manual"` | Read `Location` + `Set-Cookie` headers, return redirect |
| Network error | Any SDK call | Let throw to ErrorBoundary |

### 7.1 `422 browser_location_change_required`

This is returned by Kratos when it needs the browser to navigate (e.g. during OIDC flows). The response body is `ErrorBrowserLocationChangeRequired`:

```ts
interface ErrorBrowserLocationChangeRequired {
  error?: ErrorGeneric;
  redirect_browser_to?: string;
}
```

In a pure SSR browser flow (no AJAX), this is uncommon but must still be handled:

```ts
if (err instanceof ResponseError && err.response.status === 422) {
  const body = await err.response.json().catch(() => ({})) as ErrorBrowserLocationChangeRequired;
  if (body.redirect_browser_to) {
    return redirect(body.redirect_browser_to);
  }
}
```

### 7.2 Privileged session error

`NeedsPrivilegedSessionError`:
```ts
interface NeedsPrivilegedSessionError {
  error?: GenericError;
  redirect_browser_to: string; // login URL with refresh=true
}
```

Returned as `403` body when settings tries to modify a protected field without a fresh enough session.

### 7.3 AAL not satisfied

`ErrorAuthenticatorAssuranceLevelNotSatisfied`:
```ts
interface ErrorAuthenticatorAssuranceLevelNotSatisfied {
  error?: GenericError;
  redirect_browser_to?: string; // login URL with aal=aal2
}
```

Returned as `403` body when a second factor is required.

## 8. WebAuthn / Passkey special handling

WebAuthn and Passkey require browser-side JavaScript execution triggered from server-rendered form nodes. This is the most complex rendering case.

### 8.1 Script loading

For any flow that contains script nodes (`node.type === "script"`), those scripts must be injected into the page. In React Router v7 / Vite, inject them via `useEffect`:

```tsx
useEffect(() => {
  const scriptNodes = flow.ui.nodes.filter(n => n.type === "script");
  scriptNodes.forEach(node => {
    const attrs = node.attributes as UiNodeScriptAttributes;
    const el = document.createElement("script");
    el.src = attrs.src;
    el.async = attrs.async;
    el.integrity = attrs.integrity;
    el.crossOrigin = attrs.crossorigin;
    el.referrerPolicy = attrs.referrerpolicy;
    el.nonce = attrs.nonce;
    el.type = attrs.type;
    document.body.appendChild(el);
    return () => document.body.removeChild(el);
  });
}, []);
```

### 8.2 Trigger wiring

After the script loads, the Ory WebAuthn functions are available on `window`. Wire up `onclickTrigger` and `onloadTrigger` on mount:

```tsx
useEffect(() => {
  // Wire onload triggers
  flow.ui.nodes.forEach(node => {
    if (node.attributes.node_type !== "input") return;
    const trigger = (node.attributes as UiNodeInputAttributes).onloadTrigger;
    if (trigger) {
      const fn = (window as any)[trigger];
      if (typeof fn === "function") fn();
    }
  });
}, []);

// For click triggers â€” attach to button's onClick:
const handleWebAuthnClick = (trigger: string) => {
  const fn = (window as any)[trigger];
  if (typeof fn === "function") fn();
};
```

### 8.3 Passkey autocomplete

The `oryPasskeyLoginAutocompleteInit` trigger node fires on page load to initialize passkey conditional UI (autofill). This must run shortly after the WebAuthn script loads.

## 9. CSRF management in SSR

### 9.1 What Kratos provides

Kratos manages CSRF entirely. The SSR app does NOT need its own CSRF middleware for Kratos form submissions (unlike the sample app, which has Hydra OAuth2 routes requiring additional CSRF).

- On browser flow init: Kratos sets an anti-CSRF cookie.
- In `flow.ui.nodes`: a hidden input in the `default` group with `name="csrf_token"` and the correct value.
- On form submission: Kratos validates the submitted `csrf_token` against the cookie.

### 9.2 SSR app responsibility

- **Forward browser cookies** to Kratos on every server-side API call.
- **Never strip** `Set-Cookie` headers when proxying or redirecting.
- **Include the `csrf_token` input** in every form submission â€” it's in `default` group as `type: "hidden"` so it renders automatically if you render all nodes.
- **Do not** attempt to generate or validate CSRF tokens in the SSR app itself for Kratos flows.

### 9.3 Cookie forwarding pattern (consolidated)

```ts
const cookie = request.headers.get("cookie") ?? undefined;
// Pass to every Kratos SDK call that operates on behalf of the browser session
await kratosInternal.getLoginFlow({ id: flowId, cookie });
await kratosInternal.toSession({ cookie });
await kratosInternal.createBrowserLogoutFlow({ cookie });
```

## 11. SDK singleton pattern

The `FrontendApi` client should be instantiated once (module-level singleton), not per request. The `cookie` parameter is passed per-call, so there is no per-request state in the SDK instance itself:

```ts
// lib/kratos.ts (server-only)
import { FrontendApi, Configuration } from "@ory/kratos-client-fetch";

export const kratosInternal = new FrontendApi(
  new Configuration({
    basePath: process.env.KRATOS_PUBLIC_URL ?? "http://localhost:4433",
  })
);

export const KRATOS_BROWSER_URL =
  process.env.KRATOS_BROWSER_URL ?? "http://localhost:4433";

export function flowInitUrl(
  flowType: "login" | "registration" | "recovery" | "verification" | "settings",
  params?: URLSearchParams
): string {
  const base = `${KRATOS_BROWSER_URL}/self-service/${flowType}/browser`;
  return params ? `${base}?${params}` : base;
}
```

## 12. Node filtering and grouping utilities

The sample app's `filterNodesByGroups` utility (from `src/pkg/ui_nodes.ts`) is a useful pattern to port. Key logic:

1. Always include `default` group nodes (for `csrf_token` and shared hidden inputs).
2. For a specific method form (e.g. password), include `default` + `password` groups.
3. For WebAuthn/Passkey, also automatically include `script` type nodes.
4. For the submit button, include only the submit node of the current method to avoid dispatching the wrong method.

Porting to TypeScript for our codebase:

```ts
import type { UiNode } from "@ory/kratos-client-fetch";

export function filterNodes(
  nodes: UiNode[],
  groups: string[],
  { includeDefault = true }: { includeDefault?: boolean } = {}
): UiNode[] {
  const allowedGroups = new Set(includeDefault ? ["default", ...groups] : groups);
  return nodes.filter(node => allowedGroups.has(node.group));
}
```

## 14. Routing architecture summary

Each route in `web/apps/accounts/app/routes/` is a React Router v7 route module:

```
routes/
  _index.tsx        â€” redirect: authenticated â†’ /settings, unauthenticated â†’ /login (Kratos init)
  login.tsx         â€” login flow renderer + loader
  registration.tsx  â€” registration flow renderer + loader
  logout.tsx        â€” logout: loader creates logout flow, redirects to logout_url
  settings.tsx      â€” settings flow renderer + loader (auth guard)
  recovery.tsx      â€” recovery flow renderer + loader
  verification.tsx  â€” verification flow renderer + loader
```

Each route module exports:
- `loader` â€” fetches Kratos flow data (server-side only)
- `default` â€” React component rendering the form

Shared utilities live in `app/lib/`:
- `kratos.ts` â€” SDK singleton + URL helpers
- `nodes.tsx` â€” `FlowNode` component (renders any `UiNode` type)
- `messages.tsx` â€” `FlowMessages` component (renders `UiText[]`)

## 15. Non-negotiable requirements (consolidated)

From `research.md` plus implementation-level findings:

1. **Dynamic rendering** â€” render all forms from `flow.ui.nodes`, never hardcode field names.
2. **All node types** â€” handle `input`, `text`, `img`, `a`, `script`, and `div` node types.
3. **All message levels** â€” render `flow.ui.messages` AND per-node `node.messages`.
4. **CSRF + cookies** â€” forward `Cookie` header on all SSR â†’ Kratos API calls; include hidden `csrf_token` input.
5. **Flow expiration** â€” detect `410` / `404`, re-initialize flow, do not replay payloads.
6. **Error flow page** â€” `/error` route fetches and renders Kratos error details.
7. **Session truth** â€” `toSession()` for route guards; never trust local state alone.
8. **`422` handling** â€” detect `browser_location_change_required`, redirect to `redirect_browser_to`.
9. **`return_to` propagation** â€” carry forward when chaining flows; only allowed values will be accepted by Kratos.
10. **Privileged session / AAL** â€” detect `403` + `redirect_browser_to` on settings; redirect cleanly.
11. **Method dispatch** â€” submit exactly the intended method group; don't double-submit method selectors.
12. **Non-enumeration** â€” neutral messaging for recovery/verification flows.
13. **WebAuthn/Passkey** â€” load script nodes with exact attributes (integrity, crossorigin); wire triggers.
14. **Browser URL vs internal URL** â€” internal for SSR â†’ Kratos; browser URL only for constructing init redirect targets.
15. **No React Router `action`** â€” Kratos form POSTs go directly to `flow.ui.action`, bypassing React Router.
